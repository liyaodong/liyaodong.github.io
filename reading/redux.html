1 Style Guide

All the style guide was generated from the official style guide.

1.1 Do not mutate state   essential

1.2 Reducers and it's sub utils Must Not Have Side Effects   essential

1.3 Do Not Put Non-Serializable Values in State or Actions, except middlewares like redux-thunk   essential

1.4 Only One Redux Store Per App   essential

1.5 Use Redux Toolkit for Writing Redux Logic   stronglyRecommended

1.6 Use Immer for Writing Immutable Updates   stronglyRecommended

1.7 Structure Files as Feature Folders or Ducks   stronglyRecommended

Ducks want you organize the redux store as a single file, single file for action/reducer/selector/thunks etc

1.8 Put as Much Logic as Possible in Reducers   stronglyRecommended

Since people know reducer logic should be immutable and reducer is more easier to test. write into some other place like action creator may forgot state is not mutable

1.9 Reducers Should Own the State Shape   stronglyRecommended

Don't trust the action.payload and decide the state by this payload. bad example: return { ...state, ...action.payload }

1.10 Name State Slices Based On the Stored Data   stronglyRecommended

The slice name should be some thing like `user` `posts` rather than `userReducer` `postsReducer`

1.11 Treat Reducers as State Machines   stronglyRecommended

If the state can depends on current state, then better do it like this. don't rely on all the things from actions. don't write two many if-else outside of reducer.

TS feature can help on this: Discriminated Unions The post Make Impossible States Impossible worth a try 

import {
  FETCH_USER,
  // ...
} from './actions'

const IDLE_STATUS = 'idle';
const LOADING_STATUS = 'loading';
const SUCCESS_STATUS = 'success';
const FAILURE_STATUS = 'failure';

const fetchIdleUserReducer = (state, action) => {
  // state.status is "idle"
  switch (action.type) {
  case FETCH_USER:
    return {
      ...state,
      status: LOADING_STATUS
    }
  }
  default:
  return state;
}
}

// ... other reducers

const fetchUserReducer = (state, action) => {
  switch (state.status) {
  case IDLE_STATUS:
    return fetchIdleUserReducer(state, action);
  case LOADING_STATUS:
    return fetchLoadingUserReducer(state, action);
  case SUCCESS_STATUS:
    return fetchSuccessUserReducer(state, action);
  case FAILURE_STATUS:
    return fetchFailureUserReducer(state, action);
  default:
    // this should never be reached
    return state;
  }
}

1.12 Normalize Complex Nested/Relational State   stronglyRecommended

1.13 Model Actions as Events, Not Setters   stronglyRecommended

Setters usually more than events, i.e. `food/orderAdded, pizza: 1, coke: 1` vs `orders/setPizzasOrdere, 1` `orders/setCokesOrdered, 1`

1.14 Write Meaningful Action Names   stronglyRecommended

Actions should be written with meaningful, informative, descriptive type fields. This could help on debugging and code readability.

1.15 Allow Many Reducers to Respond to the Same Action   stronglyRecommended

Smaller reducer lead to smaller function, by allow many reducer listen on single action, this would help you lead smaller logic.

1.16 Avoid Dispatching Many Actions Sequentially   stronglyRecommended

Better to use single "event" like action instead, because multiple dispatch may cause unnecessary UI update. if you really can't avoid this, try the batch from react-redux
